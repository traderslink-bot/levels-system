# Trading Journal Execution Context Plan

Date: 2026-05-27 America/Toronto
Branch: `plan/journal-execution-context`
Scope: planning only

## 1. Executive Summary

The level system now has the foundation needed to support professional trading-journal execution analysis:

- no-lookahead candle-close filtering
- formal market-structure `asOfTimestamp` protection
- runtime parity safety for support/resistance output
- optional `enrichedAnalysis` metadata on runtime zones
- pure Market Context Classifier
- optional market-context integration helpers
- `SessionMarketFacts`
- `VolumeMarketFacts`
- `VolumeShelf` detector
- `MarketContextFactsBundle`

The next major product layer should be an execution context system for the trading journal.

The purpose is to evaluate real buys and sells against the structure that existed at the time of each execution. This is not a trade recommendation engine. It is a factual and interpretive review layer that helps explain whether a trader bought near support, chased into resistance, added after confirmation, sold into strength, held after failure, or gave back profit after structure shifted.

This plan is documentation only. It does not authorize production code changes by itself.

## 2. Core Principle

Every execution must be analyzed using only information available at or before the execution timestamp.

No future candles. No still-forming candles. No hindsight levels. No post-trade structure backfill.

The journal should preserve two separate views:

1. Execution-time context: what the trader could have known at that moment.
2. Post-trade review context: what happened after the execution.

These views must not be mixed.

## 3. What The Journal Context Layer Should Do

For each execution, the system should answer factual questions first:

- What was the execution price?
- Was it a buy or sell?
- What support was nearest at that time?
- What resistance was nearest at that time?
- Was price near VWAP as a fact?
- Was price near high of day or low of day?
- Was price in a premarket/day-trade/runner/swing/chop context?
- Was volume elevated, drying up, accelerating, or exhausting?
- Was there a nearby invalidation level?
- Was there a nearby target/resistance level?

Then the system can produce review labels:

- bought near support
- bought into resistance
- added after confirmation
- added into extension
- sold into strength
- sold into support
- held after failed breakout
- held after VWAP loss
- reduced risk at a logical level
- chased far from base

Those review labels should be generated by a later interpretation layer, not by the raw facts layer.

## 7. Risk Context Model

Suggested type:

```ts
// 2026-05-27 America/Toronto
export type ExecutionRiskContext = {
  nearestInvalidationLevel: number | null;
  distanceToInvalidationPct: number | null;
  nearestTargetLevel: number | null;
  distanceToTargetPct: number | null;
  riskRewardToNearestTarget?: number | null;
  hasDefinedRisk: boolean;
  reason: string;
};
```

For a long trade, likely invalidation may be nearest support below entry. Likely target may be nearest resistance above entry.

For a sell execution, context should explain whether the sale occurred near resistance, into extension, after support failure, or in the middle of the range.

## 8. Inputs Needed

The execution context builder should eventually accept:

- execution object
- closed 5m candles
- optional 1m candles for partial-derived future work
- optional 15m candles later
- optional daily/4h candles
- current or as-of `LevelEngineOutput`
- session facts
- volume facts
- volume shelves
- market context profile
- facts bundle
- previous close
- optional explicit catalyst/news timestamp

The first implementation should prefer already-built facts and already-built level output where possible.

## 9. No-Lookahead Contract

Mandatory:

- all candles must be filtered by candle-close semantics
- all facts must use `asOfTimestamp`
- levels must be generated from an as-of-safe candle window when used for historical/replay analysis
- execution-time context must not use future candles after the execution
- current execution candle is excluded unless it is closed or partial-derived from lower-timeframe closed candles

Tests must prove:

- appending a future candle after the execution timestamp does not change execution context
- a 5m candle starting at 09:30 is not eligible for a 09:33 execution
- formal structure and market context do not backfill future BOS/CHOCH into earlier executions

## 10. Behavior Pattern Layer Later

The execution context snapshot should not directly say the trader made a bad decision. It should provide the evidence needed for later pattern detection.

Later behavior patterns may include:

- chase entry
- support entry
- breakout confirmation entry
- add into resistance
- average down after support failure
- sell into resistance
- panic sell into support
- held after failed breakout
- held after VWAP loss
- gave back profit after resistance rejection

This keeps the architecture clean:

```text
Facts layer -> execution context -> behavior detection -> scoring -> coaching
```

## 11. Entry Quality Inputs

Later entry scoring should consider:

- distance from nearest support
- distance to nearest resistance
- distance from VWAP as a fact
- market context profile
- runner phase
- volume state
- extension risk
- whether entry had nearby invalidation
- whether entry occurred after confirmation or before confirmation

Entry scoring is not part of the first implementation.

## 12. Add Quality Inputs

Later add-quality analysis should consider:

- whether structure improved before the add
- whether price was closer to support or resistance
- whether the add occurred after confirmation
- whether the add occurred after a failed breakout
- whether the add increased exposure near resistance
- whether the add raised average price into extension

Add scoring is not part of the first implementation.

## 13. Exit Quality Inputs

Later exit analysis should consider:

- whether the sell occurred into resistance
- whether the sell occurred near support
- whether the sell occurred after a failed breakout
- whether the sell reduced risk after structure broke
- whether the trader held too much after VWAP loss or support loss
- whether the sell captured extension

Exit scoring is not part of the first implementation.

## 14. Giveback Analysis Later

The journal should eventually compute:

- max open profit
- realized profit
- giveback amount
- giveback percent of max profit
- structure event tied to giveback
- whether giveback followed a resistance rejection, VWAP loss, support break, or failed breakout

Giveback analysis is not part of the first implementation.

## 16. First Implementation Phase

The first implementation should be small:

- define execution context types
- add pure helper to find nearest support/resistance from a supplied `LevelEngineOutput`
- add pure helper to build an execution snapshot from already-supplied facts
- no LevelEngine calls inside the builder
- no network calls
- no Date.now
- no behavior scoring
- no coaching text
- no alerts
- no monitoring integration

This keeps the first implementation deterministic and testable.

## 17. Tests Required For First Implementation

Tests should prove:

- nearest support below execution is found
- nearest resistance above execution is found
- extension levels can be used as target/context levels
- support/resistance distance percentages are correct
- trade location label is factual and deterministic
- risk context identifies invalidation and target for a long buy
- sell execution near resistance is identified factually
- supplied session facts are copied, not mutated
- supplied volume facts are copied, not mutated
- supplied volume shelves remain facts-only
- no `LevelEngineOutput` mutation
- old/default `LevelEngine` behavior unchanged
- no-lookahead safety flags are present

## 18. Non-Goals

- coaching output
- P/L calculations
- giveback analysis
- behavior scoring
- alert changes
- monitoring changes
- LevelEngine integration
- Discord messages
- automatic journal ingestion
- brokerage import logic
- chart rendering
